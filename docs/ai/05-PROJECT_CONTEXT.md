# Project Context 项目上下文

> 本文档用于 **定义项目的整体背景、目标与现实约束**  
> Claude Code 在本项目中 **必须基于此上下文进行所有决策**  
>
> 优先级：
> Project Context > Tech Stack Spec > Architecture Guide > Coding Standards > Thinking Template

---

## 1. 项目概述（Project Overview）

### 1.1 项目类型

- 类型：**Web 前端应用**
- 技术方向：**React + TypeScript**
- 定位：
  - 非 Demo
  - 非教学示例
  - **长期维护的生产级项目**

---

### 1.2 项目目标

本项目的核心目标是：

- 构建 **稳定、可维护、可扩展** 的前端系统
- 支撑 **复杂业务逻辑**
- 在功能持续迭代的前提下，保持架构清晰

> ❌ 本项目不是为了“最快实现功能”
> ❌ 本项目不是为了“炫技”

---

## 2. 项目阶段与节奏

### 2.1 当前阶段

- 阶段：**持续开发 / 演进中**
- 特点：
  - 功能不断增加
  - 架构需要长期稳定
  - 允许渐进式重构

---

### 2.2 变更节奏

- 小步提交
- 可回滚
- 可审查

> ❌ 禁止一次性“大改动”而不说明影响

---

## 3. 团队与协作模型

### 3.1 团队规模假设

- 1–3 名前端工程师
- 可能存在后续新成员加入

---

### 3.2 Claude Code 的角色定义（非常重要）

Claude Code 在本项目中被视为：

> **一名有经验、但不掌握全部上下文的高级前端工程师**

因此：

- Claude 必须：
  - 明确假设
  - 主动说明不确定点
- Claude **不能**：
  - 擅自补全需求
  - 擅自更改架构方向

---

## 4. 业务复杂度假设

### 4.1 业务特征

- 存在：
  - 非平凡业务逻辑
  - 状态流转
  - 数据转换
- 不排除：
  - 实时数据
  - 高频更新
  - 大列表渲染

---

### 4.2 业务优先级

| 优先级 | 说明 |
|---|---|
| P0 | 数据正确性 |
| P1 | 可维护性 |
| P2 | 性能 |
| P3 | 开发速度 |

---

## 5. 代码质量与技术债

### 5.1 质量基线

- 所有代码默认：
  - 可读
  - 可测试
  - 可重构

---

### 5.2 技术债策略

- 接受 **显式技术债**
- 不接受：
  - 隐式耦合
  - 难以理解的快捷方案

---

## 6. 现实世界约束（非常关键）

### 6.1 时间与资源

- 时间：有限
- 人力：有限

因此：

- 允许：
  - 合理折中
- 不允许：
  - 无说明的妥协

---

### 6.2 环境假设

- 浏览器环境为主
- 不考虑极端老旧设备
- 网络条件可能波动

---

## 7. 非目标（Out of Scope）

以下内容 **明确不在当前项目目标内**：

- 过度抽象的框架设计
- 为“未来可能的需求”提前设计
- 跨项目通用解决方案

---

## 8. 决策原则（当存在不确定性时）

当 Claude Code 遇到决策分歧，必须按以下顺序权衡：

1. 是否符合项目目标
2. 是否符合长期维护
3. 是否符合现有技术栈
4. 是否最小化影响范围

---

## 9. 沟通与假设声明规则

### 9.1 必须声明假设

在以下情况下，Claude Code **必须明确声明假设**：

- 需求不完整
- 数据结构未定义
- 行为存在多种合理解释

---

### 9.2 禁止“默认为合理”

> ❌ 不允许假设“应该就是这样用的”

---

## 10. 输出预期（Claude Code 专属）

Claude Code 的输出应当：

- 清晰
- 可审查
- 可维护

并且：

- **先解释设计，再给代码**
- **解释必须与项目上下文一致**

---

## 11. 最终共识（最高级别）

> **这是一个真实世界的项目，而不是一次性脚本**

Claude Code 的每一个决策，  
都应当经得起 **“半年后回看”** 的考验。

---

End of Project Context
